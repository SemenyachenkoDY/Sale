# Создание шаблонов задач с использованием контекста Airflow. Привести пример


## 1. Что такое Apache Airflow и зачем нужны шаблоны?

Apache Airflow — это платформа для программного создания, планирования и мониторинга рабочих процессов (пайплайнов данных). Каждый пайплайн представлен в виде **DAG** (Directed Acyclic Graph, направленный ациклический граф) — набора задач с определёнными зависимостями.

При построении реальных пайплайнов возникает одна принципиальная проблема: задачи нужно делать **динамическими**. Например, задача по выгрузке данных должна сама знать, за какую дату она запущена, или задача по созданию отчёта должна формировать правильное имя файла с датой. Жёстко прописывать конкретные значения — плохая практика, потому что тогда DAG перестаёт быть переиспользуемым.

Именно для этого Airflow предоставляет механизм **шаблонизации на основе контекста**.

---

## 2. Что такое контекст Airflow?

**Контекст (execution context)** — это специальный словарь, который Airflow автоматически формирует и передаёт каждой задаче в момент её выполнения. Этот словарь содержит всю актуальную информацию о текущем запуске: время, дату, идентификаторы, параметры, ссылки на объекты DAG и TaskInstance.

Контекст формируется непосредственно перед стартом задачи и является **уникальным для каждого запуска** — то есть каждый раз, когда Airflow запускает задачу по расписанию, он создаёт новый экземпляр контекста с актуальными значениями.

> **Важно:** контекст существует **только во время выполнения** задачи. На этапе написания кода DAG (при его парсинге планировщиком) контекст ещё не доступен.

---

## 3. Два способа использования контекста

Airflow предоставляет два принципиально разных механизма доступа к контексту:

### 3.1 Jinja2-шаблоны (декларативный способ)

Первый способ — использование шаблонизатора **Jinja2**, который встроен в Airflow. Jinja2 позволяет вставлять переменные контекста прямо в строковые параметры операторов с помощью двойных фигурных скобок: `{{ имя_переменной }}`.

Airflow автоматически обнаруживает такие шаблоны в параметрах и **рендерит** (вычисляет и подставляет) их перед запуском задачи. Этот процесс называется **template rendering** — замена шаблонных выражений на реальные значения.

Этот способ не требует написания Python-кода и идеален для простых подстановок в команды, SQL-запросы, пути к файлам и конфигурации.

### 3.2 Прямой доступ через Python (программный способ)

Второй способ — получение полного словаря контекста внутри Python-функций. Когда задача реализована как Python-функция (через `PythonOperator`), Airflow может передать ей весь контекст в виде именованных аргументов. Это даёт полный программный доступ ко всем объектам Airflow.

## 4. Структура контекста: ключевые переменные

Контекст содержит десятки переменных. Ниже представлены наиболее важные из них.

### Переменные времени и дат

| Переменная | Описание |
|---|---|
| `execution_date` | Логическая дата запуска задачи (объект `pendulum.DateTime`) |
| `ds` | Строка `execution_date` в формате `YYYY-MM-DD` |
| `ds_nodash` | Строка `execution_date` в формате `YYYYMMDD` |
| `ts` | Полная временная метка в формате ISO 8601 |
| `next_ds` | Дата следующего интервала расписания |
| `prev_ds` | Дата предыдущего интервала расписания |
| `data_interval_start` | Начало интервала данных (Airflow 2.x) |
| `data_interval_end` | Конец интервала данных (Airflow 2.x) |

**`execution_date`** — центральная концепция Airflow. Это **логическая** дата запуска задачи, а не реальное время старта. Например, DAG с ежедневным расписанием, запущенный в понедельник, будет иметь `execution_date` равную воскресенью — то есть дате завершившегося интервала.

**`ds`** — самая часто используемая переменная в шаблонах, потому что большинство файлов и запросов оперируют именно датой в строковом виде.

**`next_ds`** и **`prev_ds`** — очень полезны для задач, которые обрабатывают данные за интервал между двумя датами — например, «дай мне все записи от `prev_ds` до `ds`».

### Переменные идентификации

| Переменная | Описание |
|---|---|
| `run_id` | Уникальный идентификатор текущего DAG Run |
| `dag` | Ссылка на объект текущего DAG |
| `task` | Ссылка на объект текущей задачи |
| `task_instance` / `ti` | Ссылка на объект TaskInstance |

**`task_instance`** — один из самых важных объектов контекста, потому что именно через него осуществляется обмен данными между задачами (механизм **XCom**).

### Пользовательские параметры

| Переменная | Описание |
|---|---|
| `params` | Словарь с пользовательскими параметрами DAG |
| `dag_run` | Объект текущего DAG Run (статус, конфигурация) |
| `conf` | Конфигурация Airflow (`airflow.cfg`) |

**`params`** — словарь с пользовательскими параметрами, которые передаются в DAG при его определении или при ручном запуске. Это позволяет делать DAG конфигурируемым без изменения кода.

## 5. Как шаблоны рендерятся: механизм template_fields

Не каждый параметр оператора автоматически рендерится как шаблон. В Airflow каждый оператор явно объявляет список параметров, которые поддерживают шаблонизацию. Этот список называется **`template_fields`**.

Например, `BashOperator` объявляет поле `bash_command` как шаблонное — это значит, что Airflow обработает Jinja2-выражения в строке команды. При этом поле `retries` шаблонным не является и попытка вставить туда `{{ ds }}` не сработает.

**Порядок работы механизма:**

1. Airflow получает задачу для выполнения
2. Формирует словарь контекста с актуальными значениями
3. Проходит по всем полям, перечисленным в `template_fields`
4. Для каждого поля запускает Jinja2-рендеринг с подстановкой переменных контекста
5. Только после этого запускает метод `execute()` оператора

Кроме строк, Airflow также поддерживает шаблонизацию **файлов** — если значение параметра является путём к файлу с расширением из списка `template_ext`, Airflow прочитает этот файл и отрендерит его содержимое как Jinja2-шаблон. Это особенно удобно для SQL-запросов, хранящихся в отдельных `.sql`-файлах.

## 6. Механизм XCom как расширение контекста

**XCom (Cross-Communication)** — это встроенный механизм Airflow для передачи небольших данных между задачами. Он тесно связан с контекстом, потому что доступ к XCom осуществляется через объект `task_instance`.

- Задача-отправитель **публикует** (push) значение с ключом в хранилище XCom
- Задача-получатель **забирает** (pull) это значение, указывая `task_id` отправителя и ключ

XCom хранится в метабазе Airflow и персистентен — данные сохраняются между запусками и доступны для повторного чтения.

> **Ограничение:** XCom предназначен только для **небольших объёмов данных** (строки, числа, небольшие словари). Для передачи больших датасетов используют внешние хранилища (S3, базы данных), а через XCom передают лишь путь к данным.

## 7. Концепция шаблонов задач (Task Templates)

**Шаблон задачи** — это задача (или группа задач), параметры которой полностью определяются через контекст и параметры, а не через жёстко заданные значения. Такая задача является **универсальной и переиспользуемой**.

### Главные принципы проектирования шаблонных задач

**Идемпотентность** — задача, запущенная повторно за тот же период, должна давать тот же результат и не создавать дубликатов. Использование `{{ ds }}` в условиях запроса обеспечивает это: задача всегда работает с конкретным, однозначно определённым периодом.

**Атомарность** — каждая задача должна выполнять одну логическую операцию и либо завершаться успешно целиком, либо не выполняться вовсе. Это упрощает обработку ошибок и повторные запуски.

**Параметризация через контекст** — все изменяемые значения (даты, пути, имена) должны поступать из контекста, а не быть зашиты в код. Это делает задачу применимой для любой даты запуска.

## 8. Переиспользование шаблонов: TaskGroup и Dynamic Task Mapping

Airflow предоставляет несколько механизмов для группировки и переиспользования наборов задач:

### TaskGroup

**TaskGroup** — логическая группировка задач внутри одного DAG для визуального упорядочивания в интерфейсе. Все задачи в группе по‑прежнему являются частью одного DAG и имеют доступ к одному контексту. Это современный и рекомендуемый способ структурирования сложных DAG.

### SubDAG (устаревший подход)

**SubDAG** — отдельный дочерний DAG, встраиваемый в родительский. Сейчас считается антипаттерном из-за проблем с производительностью и управлением состоянием. Вместо него рекомендуется использовать TaskGroup.

### Dynamic Task Mapping (Airflow 2.3+)

**Dynamic Task Mapping** — современный механизм, позволяющий динамически создавать множество экземпляров одной задачи на основе списка входных данных. Каждый экземпляр получает свой контекст, дополненный индексом и значением из списка. Это позволяет создавать подлинно параметризованные шаблоны, где количество задач определяется **во время выполнения**, а не при написании DAG.

## 9. Jinja2 в контексте Airflow: расширенные возможности

Помимо простой подстановки переменных, Jinja2 в Airflow поддерживает:

### Фильтры

Функции преобразования значений прямо в шаблоне. Например, можно отформатировать дату в нужный формат или преобразовать строку к верхнему регистру.

### Условные выражения

Можно вставлять `if/else` конструкции прямо в шаблон. Это позволяет, например, использовать разные пути к файлам для разных сред (production/staging) на основе параметра.

### Макросы Airflow

Airflow добавляет в окружение Jinja2 набор готовых вспомогательных функций (`macros`). Это функции для работы с датами, форматирование временных меток и другие утилиты.

### Пользовательские макросы

В DAG можно определить собственные функции и передать их в Jinja2-окружение через параметр `user_defined_macros`. Это позволяет использовать произвольную логику прямо в шаблонах без написания кода в теле задачи.
